## Dependency notes

- **001 is the safety net.** It adds the first tests for `monte-carlo.ts` and
  the run-out metric — the money path, currently untested. 004, 005, and 006 all
  change code that 001 covers; landing 001 first means their refactors are
  caught by tests rather than eyeballed.
- **002 and 003 are independent** and can be done in any order, any time. They
  touch no logic.
- **004 depends on 001** — the percentile regression test in 001 is what proves
  the O(n) rewrite produces identical output.
- **005 depends on 001** — it edits `calculateRunOutChance`, which 001 exports
  and tests. 005 will also update 001's run-out fixtures (it changes that
  function's contract); this is expected.
- **006 depends on 001** for the safety net, and is the highest-impact but
  highest-effort/risk plan. It needs an external mortality-rate dataset; it has
  a hard STOP if that data can't be obtained — do not fabricate actuarial
  numbers.

## Recommended sequencing

Fast confidence-builders first (002, 003), then the safety net (001), then the
refactors that depend on it (004, 005), then the big correctness fix (006).
002/003 can also be parallelised with 001 since they don't overlap.

## Findings considered and rejected

- **Security / dependency advisories** (`npm audit`: 20 issues): all in dev-only
  dependencies (vitest→jsdom→ws, lint-staged→yaml/micromatch). No runtime path —
  this is a static client-only app with no network, no persistence, and no
  user-supplied strings rendered as HTML. Upstream reports "no fix available".
  Re-check at the next major dependency bump; not worth a plan now.
- **No ESLint; `npm test` watch mode; `posttest` reformats the repo**: minor DX
  warts. `tsc --noEmit` + Prettier already cover most of what ESLint would catch
  on a codebase this size. Not worth a plan.
- **`DEBT-01`: `pickFromNormalDistribution` has a baked-in `/100` that callers
  compensate for inconsistently** (`utils.ts:17`): genuinely confusing, but
  mathematically correct today and the RNG is used in hot paths with thin tests
  — the risk of "tidying" it outweighs the benefit until 001's tests exist.
  Revisit after 001 if desired.
